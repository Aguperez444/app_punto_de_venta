Revisión integral del proyecto: APP_FARMACIA_HEX
Fecha: 2025-10-14 16:23

Resumen ejecutivo
- El proyecto presenta una arquitectura hexagonal razonable (domain, application, infrastructure, UI) con separación de capas y uso de UoW y repositorios.
- Hay buenas prácticas en Value Objects (Price, Stock) y entidades (Producto, Venta, DetalleVenta), aunque faltan algunas validaciones y tests.
- La capa de infraestructura con SQLAlchemy es funcional, pero hay detalles a mejorar en transacciones, logging, N+1, manejo de errores y naming consistente.
- La UI con Tkinter está estructurada en controladores y vistas, pero faltan validaciones consistentes, mensajes de error homogéneos, y una mejor separación de responsabilidades.
- Documentación y pruebas son escasas; sería valioso un plan de testing, README más completo y automatización básica.

Detalles por capa

Capa Domain
- app/domain/models/Producto.py
  - Pros: Value Objects Price y Stock integrados; método factory from_string centraliza validación/parsing.
  - Mejoras:
    1. Validar nombre, código y código de barras (no vacíos, trims, longitudes máximas).
    2. Disminuir_stock: evitar stock negativo. Si Stock.value - amount < 0 → lanzar DomainValidationError específico.
    3. Considerar __repr__ y __eq__ para facilitar debugging y testing.
    4. Propiedades solo de lectura salvo id_producto; evaluar inmutabilidad parcial y métodos para cambio controlado (p. ej. cambiar_precio, actualizar_detalle) para encapsular reglas.

- app/domain/models/price.py
  - Pros: Value object inmutable con validación de negativo; formateo amigable en __str__.
  - Mejoras:
    1. Eliminar print(normalized).
    2. Manejar separadores de miles/decimales de forma más robusta (soporte de distintos locales), o definir explícitamente el formato esperado y documentarlo.
    3. __str__: evitar lógica manual de entero/decimal si Decimal.quantize puede simplificarlo; asegurar 2 decimales fijos con ROUND_HALF_UP.
    4. From_string: capturar excepciones y adjuntar el valor original y causa, pero sin exponer stacktrace al usuario.

- app/domain/models/stock.py
  - Pros: Value object simple con from_string y __str__.
  - Mejoras:
    1. Validar no-negativo en construcción; hoy acepta negativos si se construye con int directo.
    2. From_string: aceptar números con espacios; lanzar DomainValidationError específico (ya usa StockInvalidError).

- app/domain/models/DetalleVenta.py
  - Pros: create_new válida cantidad y precio; subtotal como Decimal con redondeo a 2.
  - Mejoras:
    1. Tipos: producto y venta opcionales; documentar lifecycle.
    2. Calcular_subtotal: usar quantize(Decimal('0.01'), rounding=ROUND_HALF_UP) en vez de round, para consistencia contable.

- app/domain/models/Venta.py (no abierto aquí, pero por contexto)
  - Confirmar: tiene nueva(date), agregar_detalle, update_price; verificar que impida ventas sin detalles y calcule total con Decimal.

Capa Application
- add_product.py
  - Pros: Caso de uso minimalista con Producto.from_string.
  - Mejoras:
    1. No hace commit; depende de que el controller use with uow(): correcto si la convención es “commit en controller”. Mantener convención uniforme.
    2. Manejo de errores: capturar y traducir errores de dominio si aplica; hoy el controller lo hace.

- register_sale.py
  - Pros: Válida existencia de producto; disminuye stock; crea Venta y Detalle; persiste venta y actualiza producto; hace commit.
  - Mejoras:
    1. Eliminar print de depuración.
    2. Consistencia de transacciones: o todos los casos de uso comitean o ninguno.
    3. Posible mejora: producto_repository.update_stock(product) en lugar de update_info_product si solo cambia stock.

- Otros casos de uso (edit./update/query): revisar que no consulten directamente infraestructura y que usen puertos. Asegurar que no haya lógica de presentación.

- Puertos (IProductoRepository, ISaleRepository, IUnitOfWork)
  - Confirmar que definan claramente contrato de contexto (enter/exit) y commit/rollback. Documentar.

Capa Infrastructure (DB/ORM)
- producto_repository_impl.py
  - Pros: Métodos útiles (filtros, ordenamientos, bulk updates). Uso de mapper ProductoMapper para traducir ORM↔dominio.
  - Mejoras:
    1. Get_* que devuelven None cuando no hay resultados: preferible devolver lista vacía para evitar cheques nulos en clientes y ser más Pythonic.
    2. Filtros: construir patrones de búsqueda con sanitización básica (escapado está cubierto por bindings, pero normalizar whitespace y lower).
    3. Bulk operations update_all_prices/update_selected_product_prices: asegurar redondeo consistente a 2 decimales (Decimal + quantize) en DB y dominio.
    4. Add_stock / set_stock: validar no-negativo y evitar condiciones de carrera (usar "UPDATE ... WHERE id=?", con verificación).
    5. Performance: agregar índices en columnas usadas en filtros (nombre, detalle, codigo, codigo_de_barras, stock).
    6. Paginación: para listas grandes, proveer métodos con limit/offset.
    7. Naming: algunos métodos mezclan retornos None/list; unificar.

- ORM y mapeos (app/infrastructure/database/sqlalchemy/orm.py)
  - Revisar: tipos de columnas apropiados (Decimal para precio, Integer para stock), constraints (UNIQUE para código/código de barras), y relaciones entre Venta/DetalleVenta/Producto.

- UoW (unit_of_work_impl.py, uow_factory.py)
  - Asegurar que el context manager maneje rollback en excepción y cierre la sesión siempre.
  - Añadir logging en begin/commit/rollback.
